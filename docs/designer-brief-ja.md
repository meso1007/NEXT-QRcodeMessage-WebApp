# OTODOKE LIFE — デザイナー向けデザイン外注ブリーフ

**文書の目的**  
本サービスの**見た目・トーン・UIの一貫性**を刷新するためのデザイン制作を依頼します。**文言・情報設計・機能仕様は現状維持**とし、レイアウト・配色・タイポグラフィ・アイコン・余白・コンポーネント表現の刷新を想定しています。

**対象URL（本番想定）**  
https://otodokelife.com/

---

## 1. プロダクト概要（変更しない前提の事実）

| 項目 | 内容 |
|------|------|
| 名称 | **OTODOKE LIFE** |
| キャッチ | 大切な人への想いを、QRコードとURLで残す無料のWebサービス |
| 主要価値 | メッセージは**サーバーに保存されず**、圧縮データがURL（ハッシュ）に埋め込まれる設計 |
| 言語 | 日本語（`lang="ja"`） |
| 技術（実装側） | Next.js（App Router）、React、Tailwind CSS、Framer Motion、GSAP 等 |

デザイン作業では**上記のプロダクト説明・法的文書の意味内容は変えない**でください。表現（フォント・色・図解）は自由に提案いただけます。

## 2. スコープと制約

### 2.1 やること（デザイン）

- 全体のビジュアルアイデンティティ（色、タイポ、角丸、影、グラデーションの方針）
- 共通パーツ（ヘッダー／フッター／ボタン／入力／モーダル／カード／FAQ 等）のUIデザイン
- **レスポンシブ**（モバイル〜デスクトップ）の画面デザイン
- **アクセシビリティ**を意識したコントラスト・タップ領域・フォーカス表現の提案
- 必要に応じて**ロゴ・アイキャッチ・OGP用ビジュアル**の刷新案（既存 `/logo/app-logo.png` 等の置き換えを想定してよい）

### 2.2 やらないこと（今回の依頼範囲外）

- 新機能の追加、画面の追加・削除
- 利用規約・プライバシーポリシー・FAQ の**条文・Q&A本文の改変**（誤字修正以外はクライアント確認が必要）
- フロントエンドの実装（別工程。ただし**実装可能なトークン・コンポーネント仕様**での納品を推奨）

### 2.3 実装時に守ってほしい挙動（デザインで触れない部分）

以下は**現行の機能として固定**です。デザインはこれらを覆い隠さない・操作不能にしないようにしてください。

- メッセージ入力 → **QRコード生成**（外部QR API利用＋フォールバック）
- 生成後の **ダウンロード／URLコピー**
- バリデーション・エラー時の**モーダル表示**（メッセージ未入力・文字数超過・QR失敗・URLコピー系）
- `/letter` での **封筒クリック → 手紙表示 → 本文の逐次表示アニメーション**（演出の「見た目」は刷新可、**操作フローは同じ**）
- レビュー依頼モーダル（Googleフォームへの外部リンク）
- Google Analytics の埋め込み（画面上は目立たせない）

## 4. ホーム（`/`）— フォーム仕様（文言・制約）

デザイン上、ラベル・プレースホルダ・ボタン文言は**原則このまま**使用してください（トーンに合わせた微調整は要相談）。

| 要素 | 仕様 |
|------|------|
| 相手のお名前（任意） | `maxLength=50`、プレースホルダ `例: お母さん` |
| あなたのお名前（任意） | `maxLength=50`、プレースホルダ `例: 花子` |
| メッセージ（必須） | `maxLength=5000`、複数行、プレースホルダは現行の長文例のまま |
| 文字数ステータス | 文字数帯ごとに色分けされた説明文＋ `n/5000文字` ＋プログレスバー |
| 主CTA | `QRコードを生成する`（未入力または超過時は無効／文言変化あり） |
| QR生成後 | 見出し、QR画像（約300px表示想定）、補足文、読み取り案内ブロック、`QRコードをダウンロード` / `URLをコピー` |

**モーダル（タイトルは固定でデザイン）**

- メッセージが必要です  
- メッセージが長すぎます（5000文字以内の案内）  
- QRコード生成に失敗しました（箇条書き＋閉じる／URLをコピー）  
- URLコピーの確認（確認／キャンセル）  
- URLをコピーしました  
- URLを手動でコピー（URLをコードブロック表示）  

**レビューモーダル**

- タイトル・説明・「レビューを書く」（外部フォーム）・「後で回答する」  
- 表示タイミングは実装依存（例：利用開始から一定時間後、QR生成フロー後など）。**邪魔にならないサイズ・閉じやすさ**を重視してください。

## 5. レター閲覧（`/letter`）— 体験フロー（固定）

1. ローディング（スピナー＋「手紙を準備しています...」）  
2. 正常時：背景演出の上に **封筒**（宛名 `○○様`、クリックで開封）  
3. 開封後：**手紙用紙**上に `{相手名}へ`、本文（改行保持）、最後に日付と `{書き手名} より`  
4. 本文は**1文字ずつ表示される演出**（現行）。デザインは「手紙らしさ」を強化してよいが、**読みやすさ**を最優先  
5. エラー時：エラーメッセージ＋「テストで確認」ボタン（開発用フォールバック。デザイン上は通常エラーUIとして扱ってよい）

**BGM**  
ローディング中に音声要素あり。ユーザー環境により自動再生がブロックされる想定。**音のON/OFFや注意書き**のUIがあると望ましい（新規提案可・実装は別途相談）。

## 6. コンポーネント単位の共通要件

- **モーダル**：オーバーレイ、閉じる（×）、Esc、背景クリックで閉じる（種類により例外あり）  
- **ボタン**：プライマリ／セカンダリ／危険（削除系は現状少ない）の区別  
- **フォーム**：フォーカスリング、エラー状態、disabled 状態が識別できること  
- **フッター**：非営利・無料の注釈、規約／プライバシー／FAQ リンク、コピーライト（「Mirai Message Association」クリックで `/about#profile`）

## 7. 現状のビジュアルメモ（参考・刷新対象）

実装では主に **ピンク〜紫〜インディゴのグラデーション**、**半透明カード＋ぼかし**、**Lucideアイコン**、**Framer Motion / GSAP の動き**が使われています。  
これは**現状の暫定デザイン**であり、ブランドとして再定義して問題ありません。

フォント：レイアウトでは Geist（Google Fonts）が読み込まれ一方、`globals.css` の `body` は Arial 指定など**一部不整合**があります。新デザインでは**日本語を含むフォント方針**を明確に指定してください。
